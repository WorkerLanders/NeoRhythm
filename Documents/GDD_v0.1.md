# NeoRhythm — 게임 기획서 초안 (GDD v0.1)

## 개요

Windows 플랫폼 대상의 횡스크롤 리듬 게임. 노트가 우→좌로 이동하며 판정선 도달 시 마우스 클릭 또는 키 입력으로 타격하는 방식. 유저와 가족의 실제 인생 경험을 바탕으로 한 스토리 모드를 핵심으로 하며, AI 음악 생성 서비스를 통해 이야기에 맞는 음악을 제작한다.

---

## 1. 게임 개요

| 항목 | 내용 |
|------|------|
| 장르 | 횡스크롤 리듬 액션 |
| 플랫폼 | Windows 64-bit (최종 배포) / Web (개발용 빌드) |
| 입력 방식 | 단일 키 입력 (기본값: 마우스 좌클릭 / 설정에서 변경 가능) |
| 시점 | 2D 사이드뷰 |
| 타겟 유저 | 리듬 게임 입문~중급자 |

## 2. 핵심 메카닉

### 2-1. 레인 구성
- 화면을 수평으로 **4레인 고정** (전 난이도 공통, 확장 없음)
- **모든 레인은 단 하나의 입력 키에 대응** — 레인 구분은 시각적 다양성 목적
- 판정선은 화면 **좌측 약 15% 지점**에 수직으로 고정
- **동시에 2개 이상의 레인에 노트가 나오는 경우 없음** (채보 설계 원칙)
- 난이도 차별화는 레인 수가 아닌 **노트 밀도 및 패턴 복잡도**로 구분

### 2-2. 입력 방식

게임 전체에서 **단 하나의 입력 키**만 사용한다.

| 항목 | 내용 |
|------|------|
| 기본 키 | 마우스 좌클릭 |
| 변경 방법 | 메인 메뉴 → 설정 → 키 바인딩 |
| 지원 입력 | 마우스 버튼 (좌/우/가운데), 키보드 임의 키 |

- 어느 레인의 노트든 설정된 키 하나로 처리
- 키 타이밍만이 판정에 영향을 미침
- 키 바인딩은 **슬롯 1개** — 동시에 하나의 키만 등록 가능
- 변경된 설정은 로컬에 저장되어 다음 세션에도 유지됨

### 2-3. 노트 종류 (기본 3종)
| 종류 | 설명 | 입력 |
|------|------|------|
| **탭 노트** | 기본 단타 노트 | 설정된 키 단타 |
| **홀드 노트** | 길게 누르는 노트 (롱노트) | 설정된 키 누름 유지 |
| **슬라이드 노트** | 연속 연타 (드럼롤 느낌) | 설정된 키 연속 입력 (BPM 기반 필요 횟수 자동 계산) |

### 2-4. 노트 흐름
- 노트 생성 위치: 화면 오른쪽 바깥 (오프스크린)
- 이동 방향: 우 → 좌 (일정 속도, 노래 BPM 기반으로 스크롤 속도 조절 가능)
- 한 시점에 활성 노트는 항상 **1개 레인**에만 존재
- 판정선 통과 시 판정 발생 → 통과하면 Miss 처리

### 2-5. 슬라이드 노트 성공 조건 공식

슬라이드 구간 내 필요 클릭 수는 BPM 기반으로 자동 계산한다:

```
required_clicks = floor((duration_sec × BPM) / 60 × 0.5) + 1
```

- `duration_sec`: 슬라이드 노트 구간 길이 (초)
- `BPM`: 해당 구간의 곡 BPM
- `0.5`: 8분음표 기준 계수 (구현 시 튜닝 포인트)
- 최소 1회 보장 (`+1`)

**예시 (BPM 120, 2초 구간):** `floor(2 × 120 / 60 × 0.5) + 1 = 3회`

구간 내 `required_clicks` 이상 클릭하면 성공, 미달 시 MISS 처리.

### 2-6. 키 바인딩 시스템

플레이어는 메인 메뉴 → 설정 화면에서 입력 키를 변경할 수 있다.

**변경 흐름:**
1. 설정 화면에서 "키 바인딩" 항목 선택
2. `[ 키를 입력하세요 ]` 안내 표시
3. 원하는 키 또는 마우스 버튼을 누르면 즉시 등록
4. 저장 후 다음 곡부터 적용

**지원 입력 종류:**

| 분류 | 예시 |
|------|------|
| 마우스 버튼 | 좌클릭 (기본), 우클릭, 가운데 버튼 |
| 키보드 | 스페이스바, 영문자/숫자 키, 방향키 등 임의 키 |

**제약:**
- Esc, F-키, 시스템 예약 단축키는 바인딩 불가
- 슬롯 1개 고정 — 복수 키 동시 등록 미지원 (v1 범위 외)

## 3. 판정 시스템

| 판정 | 타이밍 오차 | 배율 |
|------|------------|------|
| **PERFECT** | ±20ms | ×1.0 |
| **GREAT** | ±50ms | ×0.8 |
| **GOOD** | ±100ms | ×0.5 |
| **MISS** | 초과 / 미입력 | ×0 (콤보 리셋) |

- 판정 결과는 판정선 위에 텍스트 + 이펙트로 즉시 표시
- 화면 하단에 **판정 히트바** (오차 분포 시각화) 표시 고려

## 4. 스코어링 & 콤보

### 만점 기준
- **만점: 1,000,000점** (곡별 노트 수와 무관하게 동일)
- 노트 1개당 기본 점수 = `1,000,000 ÷ 전체 노트 수`

### 점수 구성
| 구분 | 풀 | 설명 |
|------|----|------|
| 기본 점수 | 800,000점 | 판정 배율 적용 |
| 콤보 보너스 | 200,000점 | 최대 콤보 비율 기반, 결과 화면에서 일괄 계산 |

### 판정별 기본 점수
| 판정 | 획득 점수 |
|------|-----------|
| PERFECT | 기본 점수 × 1.0 |
| GREAT | 기본 점수 × 0.8 |
| GOOD | 기본 점수 × 0.5 |
| MISS | 0 |

### 콤보 보너스 계산식
```
콤보 보너스 = 200,000 × (최대 콤보 / 전체 노트 수)²
```
- 풀콤보에 가까울수록 보너스가 가파르게 증가하는 곡선
- 인게임 중 누적하지 않고 **결과 화면에서 일괄 산정**

**예시 (노트 100개, 최대 콤보 80):**
```
콤보 보너스 = 200,000 × (80 / 100)² = 128,000점
```

### 등급 시스템
정확도(Accuracy) = `판정 점수 합계 / (전체 노트 수 × 노트 1개 기본 점수)`

| 등급 | 정확도 조건 |
|------|------------|
| **S+** | 100% (풀퍼펙트) |
| **S** | 95% 이상 |
| **A** | 85% 이상 |
| **B** | 70% 이상 |
| **C** | 50% 이상 |
| **D** | 50% 미만 |

## 6. 유저 레벨 & HP 시스템

### 유저 레벨

**레벨 상한: 50** (추후 변경 가능)

**경험치 획득** — 곡 클리어 시 아래 공식으로 산정:
```
획득 EXP = (기본 EXP + 등급 보너스) × 난이도 배율
```

| 구분 | 값 |
|------|----|
| 기본 클리어 EXP | 100 |
| 등급 보너스 | S+: +100 / S: +80 / A: +50 / B: +20 / C·D: +0 |
| 난이도 배율 | Easy ×1.0 / Normal ×1.2 / Hard ×1.5 / Expert ×2.0 |

**예시 (Normal, S등급):** `(100 + 80) × 1.2 = 216 EXP`

**레벨업 필요 경험치**
```
레벨 N → N+1 필요 EXP = 100 × N
```

| 레벨 | 해당 레벨 필요 EXP |
|------|------------------|
| 1→2 | 100 |
| 2→3 | 200 |
| 5→6 | 500 |
| 10→11 | 1,000 |
| 49→50 | 4,900 |

---

### HP 시스템

**최대 HP 공식**
```
최대 HP = 100 + (레벨 - 1) × 10
```

| 레벨 | 최대 HP | MISS 허용 횟수 (회복 없을 시) |
|------|---------|------------------------------|
| 1 | 100 | 10회 |
| 10 | 190 | 19회 |
| 25 | 340 | 34회 |
| 50 | 590 | 59회 |

**인게임 HP 동작**

| 이벤트 | HP 변화 |
|--------|---------|
| 곡 시작 | 최대 HP 풀로 시작 |
| MISS 발생 | **-10 HP** (고정) |
| 시간 경과 | **+1 HP / 5초** (자동 회복) |
| HP 0 도달 | 즉시 곡 실패 (Arcade 모드) |

- 자동 회복은 인게임 중 실시간으로 적용
- HP는 최대 HP를 초과하여 회복되지 않음
- **연습 모드**에서는 HP 시스템 비활성화

## 8. 난이도 체계

| 레벨 | 이름 | 특징 |
|------|------|------|
| 1–3 | Easy | 단타 위주, 저밀도, 긴 노트 간격 |
| 4–6 | Normal | 홀드 노트 등장, 노트 간격 증가 |

> 전 난이도 4레인 고정. 난이도 차별화는 노트 밀도·패턴 복잡도로만 구분.

## 9. 스토리 모드 구조

> 세부 내용(에피소드·챕터·스테이지·음악 방향 등)은 별도 문서를 참고:
> **[스토리모드 기획서 → StoryMode_GDD.md](StoryMode_GDD.md)**

### 계층 구조 요약
```
게임
└── 에피소드 (4개): 그의 이야기 / 그녀의 이야기 / 고양이의 이야기 / 가족의 이야기
    └── 챕터 (여러 개 — 각 챕터는 하나의 사건을 단편적으로 표현)
        └── 스테이지 (챕터당 4개 고정 — 기·승·전·결)
            └── 음악 1곡
```

### 시각 연출 방향
- 에피소드마다 고유 색감 팔레트 및 배경 톤
- 스테이지 시작 전 짧은 텍스트 컷신으로 이야기 전달
- 노트 색상·이펙트도 스테이지 감정선에 맞춰 변경

## 10. 오디오 생성 파이프라인

### 개요
유저(개발자)의 실제 인생 경험을 기반으로 스토리라인을 구성하고, 이를 AI 음악 생성 서비스에 프롬프트로 전달해 음악을 제작하는 파이프라인.

### 파이프라인 흐름
```
[유저 인생 이야기 전달]
        ↓
[Claude: 스토리라인 & 챕터 구성]
        ↓
[Claude: 각 챕터별 음악 분위기/장르/무드 정의 → 프롬프트 생성]
        ↓
[AI 음악 생성 서비스 (Suno / Udio 등): 프롬프트 기반 음악 생성]
        ↓
[음악 파일(.mp3/.ogg) → 게임에 탑재]
        ↓
[채보 제작 (수동 or 자동 BPM 분석 기반)]
```

### AI 음악 생성 서비스 후보
| 서비스 | 특징 |
|--------|------|
| **Suno** | 가사 포함 가능, 다양한 장르, 고품질 |
| **Udio** | 세밀한 분위기 제어, 스타일 커스터마이징 |

### 채보 연동
- 생성된 음악의 BPM을 분석 (Godot 내장 or 외부 툴)
- BPM 기반으로 채보 데이터(JSON) 수작업 또는 반자동 생성

## 11. 채보 JSON 스키마

### 설계 원칙
- 시간 표현: **ms 절대값** 방식 (Godot `AudioStreamPlayer.get_playback_position() × 1000`과 직접 비교)
- 레인 배정: 채보에 포함하지 않고 **런타임 랜덤 배정** (직전 노트와 다른 레인 보장)

### 스키마 정의

```json
{
  "meta": {
    "title": "탄생, 그리고 소중함 - Stage 1",
    "artist": "AI Generated (Suno)",
    "bpm": 90,
    "duration": 142.5,
    "audio_file": "ep1_ch1_s1.ogg",
    "charter": "dev",
    "version": "1.0"
  },
  "notes": [
    {
      "id": 0,
      "type": "tap",
      "time": 1333
    },
    {
      "id": 1,
      "type": "hold",
      "time": 3000,
      "duration": 1500
    },
    {
      "id": 2,
      "type": "slide",
      "time": 6000,
      "duration": 2000
    }
  ]
}
```

### 필드 설명

**`meta`**

| 필드 | 타입 | 설명 |
|------|------|------|
| `title` | string | 곡/스테이지 제목 |
| `bpm` | number | 곡 BPM (슬라이드 필요 클릭 수 계산에 사용) |
| `duration` | number | 곡 전체 길이 (초) |
| `audio_file` | string | 오디오 파일명 |
| `charter` | string | 채보 제작자 |
| `version` | string | 채보 버전 |

**`notes` 배열**

| 필드 | 타입 | 적용 노트 | 설명 |
|------|------|-----------|------|
| `id` | int | 전체 | 노트 고유 번호 (순서 보장) |
| `type` | string | 전체 | `"tap"` / `"hold"` / `"slide"` |
| `time` | int | 전체 | 판정선 도달 목표 시각 (ms) |
| `duration` | int | hold, slide | 구간 길이 (ms) — tap은 생략 |

### 레인 랜덤 배정 (Godot 구현 참고)

```gdscript
var last_lane = 0
func get_random_lane() -> int:
    var lane = randi() % 4 + 1
    while lane == last_lane:
        lane = randi() % 4 + 1
    last_lane = lane
    return lane
```

노트 스폰 시 `get_random_lane()`을 호출하여 레인을 결정한다.

### 플랫폼 전략

PC 없이도 웹 브라우저에서 개발 빌드를 확인할 수 있도록 **이중 빌드 전략**을 채택한다.

| 구분 | 플랫폼 | 목적 |
|------|--------|------|
| 개발용 빌드 | Web (HTML5, GitHub Pages) | PC 환경 없이 어디서든 진행 상황 확인 |
| 최종 배포 빌드 | Windows 64-bit | 실제 유저 배포용 (Shipping) |

**기술적 호환성:** Godot 4의 `GL Compatibility` 렌더러는 WebGL2(웹)와 OpenGL(윈도우)를 모두 지원하므로 별도 코드 분기 없이 동일 프로젝트에서 양쪽 빌드 가능.

**주의 사항:**
- 오디오 레이턴시: 웹 빌드의 오디오 레이턴시(50–200ms)는 윈도우 네이티브(5–20ms)보다 높고 가변적. `AUDIO_OFFSET_MS` 값은 플랫폼별로 별도 캘리브레이션 필요.
- 최종 판정 튜닝(오프셋 조정)은 반드시 윈도우 빌드에서 수행.
- 세이브 데이터는 웹(IndexedDB)과 윈도우(파일 시스템) 간 공유되지 않음.
- 윈도우 export preset 및 CI 파이프라인은 M4(릴리스) 단계에서 구성.
